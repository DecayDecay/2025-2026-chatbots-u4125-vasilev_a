## 1. Источники данных

Бот получает информацию из трёх источников. Это позволяет показывать полную картину рынка: и текущие цены, и историческую аналитику, и данные о запасах предметов.

### a) Steam Community Market API

Это API торговой площадки Steam. Работает без авторизации — любой может отправлять запросы. Мы используем три эндпоинта:

- **priceoverview** — текущая цена (медианная и минимальная), объём торгов за 24 часа, количество активных листингов
- **search/render** — полный каталог предметов s&box с ценами
- **itemordershistogram** — стакан заявок (кто за сколько хочет купить и продать)

Ограничение: Steam разрешает примерно 12 запросов в минуту. Если отправлять чаще — вернёт ошибку 429 (Too Many Requests). Мы сделали ограничитель, который отправляет запросы с интервалом 5 секунд и автоматически замедляется при получении ошибки.

### b) sbox.game

Официальный сайт разработчиков s&box. На странице `sbox.game/metrics/skins` публикуется статистика продаж: выручка и количество проданных единиц для каждого скина за разные периоды (всё время, 30 дней, 7 дней, 1 день).

Проблема в том, что у этого сайта нет обычного API — данные загружаются динамически через технологию Blazor (это как серверный рендеринг от Microsoft). Поэтому обычным HTTP-запросом данные получить нельзя.

Решение: мы используем Playwright — это библиотека, которая управляет настоящим браузером (Chromium) в фоновом режиме. Браузер открывает страницу, ждёт загрузки данных, а потом мы извлекаем таблицу из HTML.

Эти данные позволяют посчитать stock (общее количество предметов в обращении) — эта информация недоступна через Steam API и является уникальным преимуществом нашего бота.

### c) PostgreSQL

Все собранные данные сохраняются в базу PostgreSQL. Доступ к базе идёт через Prisma ORM — это удобная библиотека, которая позволяет писать запросы на TypeScript вместо чистого SQL и автоматически генерирует типы для всех таблиц.

## 2. Структура базы данных

В базе 12 таблиц. Вот основные:

### Item — каталог предметов
78 скинов s&box. Для каждого хранится: название, URL иконки, тип, теги. Заполняется автоматически из Steam API.

### PriceSnapshot — снапшоты цен
Каждые 20 минут бот записывает текущую цену каждого предмета. Это позволяет строить графики и считать изменение за 24 часа.

### OrderBook — стакан заявок
Раз в час сохраняется информация о заявках на покупку и продажу: лучшая цена покупки, лучшая цена продажи, спред (разница), ликвидность.

### SboxGameStat — данные с sbox.game
Выручка и количество продаж для каждого предмета за 4 периода: всё время, 30 дней, 7 дней, 1 день.

### Пользовательские таблицы
- **Position** — позиции портфолио (что купил, по какой цене, сколько штук)
- **Alert** — ценовые алерты (уведомить когда цена выше/ниже порога)
- **Watchlist** — избранные предметы
- **Feedback** — отзывы пользователей (оценка от 1 до 5 + текст)
- **FxRate** — курсы валют (USD → KZT)
- **Settings** — настройки пользователя (выбранная валюта)

## 3. Промпты для LLM

### Основной промпт

> «Подключи бота к трём источникам данных: (1) Steam Community Market API для цен — использовать эндпоинты priceoverview, search/render и itemordershistogram без авторизации с ограничением 12 запросов в минуту; (2) sbox.game для статистики продаж — сайт на Blazor, нужен Playwright для скрапинга; (3) PostgreSQL через Prisma ORM для хранения всех данных.»

### Основные доработки

1. **Калибровка ограничителя запросов:** сначала отправляли слишком быстро и получали ошибку 429. Опытным путём нашли оптимальный интервал — 5 секунд между запросами.
2. **Скрапинг sbox.game:** сайт не имеет API, пришлось использовать виртуальный браузер (Playwright) для извлечения данных из динамически загружаемой страницы.
3. **Расчёт Market Cap:** изначально считали неправильно (на основе листингов). Исправили — теперь используем stock с sbox.game × медианная цена.
4. **Мультивалютность:** добавили конвертацию USD → KZT с автоматическим обновлением курса.

## 4. Как работает сбор данных

### Ограничитель запросов к Steam

Функция `steamGet()` — обёртка для всех запросов к Steam. Перед каждым запросом проверяет, прошло ли 5 секунд с последнего. Если нет — ждёт. Если Steam вернул ошибку 429 — увеличивает паузу вдвое и пробует снова (максимум 3 попытки).

### Скрапер sbox.game

Запускает скрытый браузер Chromium, открывает страницу `sbox.game/metrics/skins`, ждёт пока загрузятся данные (Blazor-компоненты), затем извлекает таблицу из HTML. Для каждого из 4 таймфреймов (всё время, 30 дней, 7 дней, 1 день) кликает на соответствующую вкладку и собирает данные.

### SQL-запросы для аналитики

Для сложных расчётов используется чистый SQL через Prisma: ATH (исторический максимум) через `MAX()`, изменение за 24 часа через оконную функцию `LAG()`, Market Cap через `JOIN` между таблицами PriceSnapshot и SboxGameStat.

## 6. Проблемы и решения

### a) Steam блокирует частые запросы
При отправке больше 9 запросов за 10 секунд Steam возвращает ошибку 429. Решение: поставили интервал 5 секунд между запросами и добавили автоматическое замедление при ошибках.

### b) sbox.game не имеет API
Данные на сайте загружаются динамически через Blazor Server + SignalR. Обычный HTTP-запрос получает пустую страницу. Решение: Playwright запускает реальный браузер, который ждёт загрузки данных и потом извлекает их из DOM.

### c) tsx ломает Playwright
Мы запускаем TypeScript через tsx (без компиляции). tsx добавляет свою обёртку `__name()` к функциям. Когда передаёшь функцию в Playwright для выполнения в браузере — эта обёртка вызывает ошибку, потому что в браузере её нет. Решение: передавать код в виде строки, а не функции.

### d) Неправильный Market Cap
Сначала считали Market Cap как количество листингов × цену, но это неверно — листинги это только то, что выставлено на продажу. Правильный расчёт: stock (общее количество в обращении, из sbox.game) × медианная цена.
